Smooth transitions with refs in React.js

Dec 1, 2016

Challenge

  • I want a reusable React component which works like an expandable menu button following the Google material design guidelines
  • this component should be flexible enough to allow arbitrary content to de displayed inside the button and menu area.
  • container size is not known beforehand, as the length of the button text and menu context may vary, so straightforward approach using CSS animations does not work.
  • react-css-transition-replace cannot be used, as it does not handle container width changes.

Solution

In order to use CSS transitions, we need to have fixed values for container width and height in both states: opened and closed. "Pure" React is not helpful in this case, as it knows nothing about the element sizes, but one thing we can do is to get a reference to the DOM node after an element has been mounted and access its clientWidth and clientHeight properties.

Let's assume we got these references saved to _button and _menu instance attributes of out components. In this case, getting these CSS values is as simple as having this method in your component definition:

 1 containerStyle(): CSSProperties {
 2   const { isOpen } = this.props;
 3   const button = this._button;
 4   const menu = this._menu;
 5 
 6   if (!button || !menu) {
 7     return {};
 8   }
 9 
10   if (!isOpen) {
11     return {
12       height: button.clientHeight,
13       width: button.clientWidth
14     };
15   }
16 
17   return {
18     height: menu.clientHeight + button.clientHeight,
19     width: R.max(menu.clientWidth, button.clientWidth)
20   };
21 }

Note the extra condition at lines 4-6. We're going to use these styles in the render method, and this means that it may be called when the component is not mounted yet, so refs won't be available.

Next step: let's use HOC to satisfy the "can render arbitrary content" requirement.

1 const Dropdown: ComponentClass<Openable> =
2   genericDropdown<{}>(Button, Menu);

We're passing two components which will be used to render the clickable area and the dropdown menu. These two components are fully isolated from each other.

 1 const Button: React.StatelessComponent<{}> = R.always(
 2   <div className={ styles.button }>Click Me!</div>
 3 );
 4 
 5 const Menu: React.StatelessComponent<{}> = R.always(
 6   <div className={ styles.menu }>
 7     <div className={ styles.menuItem }>Eat the mysterious pie</div>
 8     <div className={ styles.menuItem }>Cry</div>
 9     <div className={ styles.menuItem }>Drink the mysterious potion</div>
10     <div className={ styles.menuItem }>Go adventuring</div>
11   </div>
12 );

If you're confused by R.always(...) above, is a declarative shorthand for () => ... provided by the Ramda toolbelt library.

Now to the actual implementation of the HOC:

 1 function genericDropdown<Props>(
 2   Button: E<Props>,
 3   Menu: E<Props>
 4 ): ComponentClass<Props & Openable> {
 5   return createClass<(Props & Openable), DropdownState>({
 6     displayName: "ExpandableDropdown",
 7 
 8     containerStyle(): ...
 9 
10     render(): ReactElement<Props> {
11       const { isOpen, toggle } = this.props;
12 
13       return (
14         <div className={ isOpen && styles.containerOpen || styles.container }
15              style={ this.containerStyle() }
16              onClick={ toggle }>
17           <div>
18             <div className={ styles.buttonContainer }
19                  ref={ ref(this, "_button") }>
20               <Button { ...this.props } />
21             </div>
22           </div>
23           <div>
24             <div className={ styles.menuContainer }
25                  ref={ ref(this, "_menu") } >
26               <Menu { ...this.props } />
27             </div>
28           </div>
29         </div>
30       );
31     }
32   })
33 }

I've used css-modules here. It is not a requirement, normal strings as CSS classe names would do, but I do want my CS classes to be unique to reduce blog maintenance in future

Looks simple, but... doesn't work yet. If we add some debug logging to the render method and ref helper, we would see that render is called first, and only after that we get two calls to ref. The obvious reason is React requires the component to be rendered to the virtual dom before it is mounted to the actual document.

If only we could force the update after the component is mounted...

1 componentDidMount(): void {
2   this.forceUpdate();
3 }

... and yes we can.

Now the only remaining part is to add some nice styling and actual CSS transitions to the component.
